Product focus

What Trident ChatOps does

Trident ChatOps is built for NOC engineers, field technicians, and operations teams that need quick network answers without jumping between SSH terminals, monitoring dashboards, spreadsheets, ticket tools, and runbook wikis.

Natural-language device queries

Ask for interface status, BGP neighbors, OSPF state, routing tables, ping, traceroute, config backup, or device inventory in plain English.

SSH-backed diagnostics

ChatOps connects to registered Cisco, Arista, Juniper, and Linux devices, runs the correct command, and returns structured output.

Operational context in one place

Alerts, device details, runbooks, command output, and ticket context stay in the same workflow so engineers do not start from scratch.

Product workflow

From question to device evidence

The platform recognizes intent, maps the request to the right device operation, executes through SSH, formats the result, and keeps an audit trail.

1

Ask

Example: "bgp neighbors for edge-router" or "show interfaces on branch-fw-01".

2

Route

ChatOps identifies the intent, target device, vendor family, and command template.

3

Run

The platform connects to the registered device and gathers the requested output.

4

Review

Engineers see readable results, related alerts, and runbook guidance where configured.

Core functionality

Network operations built into ChatOps

The product is focused on practical NOC tasks: state checks, diagnostics, alert context, runbooks, and traceability.

Interfaces and reachability

Check interface state, descriptions, error counters, ping, traceroute, and basic path symptoms.

Routing protocols

Inspect BGP and OSPF neighbors, route tables, peer state, and session health indicators.

Config and inventory context

Keep device inventory, retrieve configuration context, and compare current state against expected operations notes.

Alerts and acknowledgements

Raise alerts for interface failures or routing-session drops, acknowledge them, and attach context to ticket workflows.

Runbook access

Give engineers the right operating procedure inside the same chat workflow.

Audit trail

Record who ran what, against which device, and when, so operational activity is reviewable.

Deployment and integrations

Designed for controlled environments

Trident ChatOps runs on-premise or in a customer-controlled VM. Device credentials and inventory stay inside the client environment.

On-premise application

FastAPI-based deployment on Linux or a controlled cloud VM, with no required external SaaS dependency for structured network queries.

Role-based access

JWT authentication and role-based controls for engineers, operators, and administrators.

Integrations

Slack, PagerDuty, Datadog/webhook inputs, Prometheus metrics, and REST API access where configured.

Product boundaries

What ChatOps is, and what it is not

The product is a force multiplier for network operations. It assists engineers with evidence and workflows; it is not an autonomous engineer and does not silently change production networks.

Does

Query devices, parse command output, surface alerts, expose runbooks, create context for tickets, and maintain an audit trail.

Does not

Resolve tickets on its own, replace engineers, or make autonomous production configuration changes.

Human approval stays central

Configuration changes require explicit human action and client-approved operating procedures.

Service layer

How operation support uses the product

After ChatOps is deployed and devices are registered, support workflows can include monitoring response, helpdesk coordination, cloud operations, staff augmentation, and incident response.

Monitoring response

Engineers review ChatOps alerts, device context, and runbooks before taking action.

Helpdesk support

Human teams own ticket response; ChatOps provides technical context when network diagnostics are relevant.

Incident coordination

Critical incidents receive structured summaries, supporting evidence, and follow-up actions.
